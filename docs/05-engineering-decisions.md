# V. **Engineering Decisions**

> *This document describes the engineering decisions that define the structure and technology stack of the service layer.*

| Decision | Rationale |
|----------|-----------|
| **Principle of Least Privilege** | For each infrastructure service — database, object storage, and message broker — dedicated credentials are created with the minimum set of permissions required by the component that uses them. For example, the worker's database credentials permit only the operations the worker actually performs, and do not include the ability to drop tables or modify schema. This limits the impact of a credential leak: a compromised secret exposes only the operations that secret was permitted to perform, rather than granting unrestricted access to the underlying infrastructure. |
| **Separation of inference and post-processing workers** | The two heavy operations in the system have fundamentally different resource profiles. Inference workers load large ML model weights into GPU memory on startup and hold them resident for the lifetime of the process — this is a deliberate trade-off that eliminates repeated weight loading overhead at the cost of permanent VRAM allocation. Post-processing workers (e.g. video rendering) are CPU-bound and require no ML dependencies. Running a single worker type for both operations would force every worker process to load model weights into memory regardless of whether it will ever execute inference — wasteful and unnecessary. Separating them into two distinct worker types ensures that model weights are loaded only in processes that actually perform inference. |
| **Pre-creation deduplication via content-addressed storage keys** | Storing duplicate artifacts in object storage — for example, when a client re-uploads the same video or re-submits the same configuration — wastes storage, repeats expensive inference or rendering operations, and accumulates redundant records. To avoid this, every resource is identified by a deterministic key derived from its content: video ingestion is keyed on the content hash of the uploaded file, while estimation and visualization jobs are keyed on a canonical hash of their input parameters. Before creating a new record, the system checks whether a matching key already exists. If a match is found, the existing record is returned immediately without re-executing the operation or writing a duplicate artifact. |
| **Environment parity via containerization** | The service must be runnable both locally for development and in a fully containerized environment for deployment. All infrastructure services — database, message broker, object storage — are provisioned via Docker Compose, ensuring that the local development environment is structurally identical to the production one. This eliminates environment-specific bugs and makes the full system reproducible on any machine with Docker installed. |
| **Video validation before storing** | The contract states that *«the client interacts with the system exclusively through network requests»*, which means the service is the first point of contact for all incoming data. Accepting arbitrary files without validation creates security risks (malware, non-video content) and wastes storage and compute resources. By validating the file (checking MIME type, file signature, duration, resolution) before persisting it to object storage, we reject invalid inputs early, protect downstream components, and ensure that only valid videos enter the processing pipeline. |
| **Video uploaded via request body, not via direct object storage URL** | The video validation described above requires the service layer to inspect incoming files. If presigned URLs were used for direct upload to object storage, this validation would be bypassed, along with rate limiting and upload tracking. By requiring clients to upload videos through the API (in the request body), the service maintains control over the ingestion pipeline, can validate and sanitize inputs, and can enforce authentication and authorization before any data reaches object storage. |
| **Streaming video processing** | Both inference and post-processing operations load video in fixed-size frame batches rather than reading the full file into memory at once. Holding an entire video in memory simultaneously alongside its accumulated estimation results is resource-intensive and creates a risk of out-of-memory crashes, particularly under concurrent load. Batch-wise processing bounds memory consumption to the size of a single chunk: frames are loaded, processed, and released before the next batch is fetched, keeping peak memory usage predictable and independent of video duration. [^1]: In an ideal architecture, the generated artifacts — serialized pose estimations and rendered visualization videos — would be streamed directly into object storage tensor-by-tensor or frame-by-frame as they are produced by the workers, completely bypassing local disk accumulation. However, the S3 API (and compatible object storage protocols) imposes fundamental constraints on streaming uploads. A standard `PUT` operation requires the `Content-Length` header to be known in advance, which is impossible for dynamically generated MP4 videos (where the final bitrate and the trailing `moov` atom size are only known at the end of encoding) or variable-length `safetensors` archives. While S3's Multipart Upload API theoretically allows streaming data in chunks, it enforces strict limitations: a minimum part size of 5 MB, a maximum of 10,000 parts, and the requirement to explicitly finalize the upload with a manifest of all part ETags. Implementing this buffering and multipart orchestration logic directly within the inference and rendering workers would significantly increase their complexity, couple them tightly to the storage backend, and still require holding multi-megabyte buffers in memory. Therefore, the current design opts for a simpler, more robust approach: workers serialize the complete artifact to a temporary local volume and delegate the upload to the service layer upon task completion.
