# IX. **Dependencies Overview**

> *This document describes the Python dependency stack, infrastructure services, system requirements, and model weights required to run the Human Pose Estimation Service.*

## System Dependencies

| Dependency | Purpose in This Project | Why Not Alternatives? |
|------------|-------------------------|----------------------|
| [**Python**](https://www.python.org/) | Primary language for all project source code. | 1. Python is the de facto standard for machine learning and computer vision. Its ecosystem is significantly more mature than those of other languages, providing a comprehensive suite of production-ready libraries.<br>2. The inference core (`ensam3d_inference`) and the entire underlying ML stack (PyTorch, CUDA bindings, `safetensors`, Vision Transformer backbones) are natively implemented in Python. Writing the service layer in a different language (e.g., Go, Rust, Java) would force the introduction of an inter-process communication boundary (gRPC, FFI, or HTTP sidecar) between the API and the inference pipeline — adding serialization overhead, operational complexity, and a split-language maintenance burden. A unified Python stack keeps the service layer and the inference core in the same runtime, enabling direct in-process calls and shared type contracts. |
| [**CPython**](https://github.com/python/cpython) | Reference Python virtual machine used for executing the source code. | CPython is the de facto standard implementation of the Python language. That's why the scientific Python stack is tightly coupled to CPython's C-API and native extension model. Alternative implementations (PyPy, GraalPy) lack full compatibility with compiled libraries, and offer no practical speedup since heavy computation already runs in optimized C/CUDA kernels. Experimental variants (freethreading, JIT) are excluded because the dependency graph is validated and locked to CPython 3.12, where these features aren't supported. |
| [**Pixi**](https://pixi.sh/latest/) | Package manager for creating reproducible environments. | 1. Unlike Uv, Pixi supports Conda packages, which are required for fully reproducible environments.<br>2. Unlike Miniconda/Micromamba, Pixi resolves mixed Conda/PyPI dependency graphs into a single deterministic lockfile. |
| [**python-dotenv**](https://github.com/theskumar/python-dotenv) | Loads environment variables from `.env` files into `os.environ` at process startup, enforcing the configuration-code separation principle (12-factor app). | 1. Manual `export` of environment variables is error-prone and does not scale across developers or CI runners.<br>2. `python-dotenv` has become the de facto standard library for this role in the Python ecosystem. Virtually all other frameworks and tools that need to interact with `.env` files implicitly rely on it under the hood, making it the most interoperable and battle-tested choice. |
| [**just**](https://github.com/casey/just) | Command runner used to store and execute long, frequently invoked developer workflows via a single versioned `Justfile`. | **Make** is historically a *build system*, not a command runner. Its implicit file-target semantics (where a recipe named `test` is skipped if a file named `test` exists), the need for `.PHONY` declarations, and quirky variable escaping (`$$`) introduce friction and hard-to-debug failures when used purely to alias shell commands. |
| **NVIDIA GPU + CUDA-compatible Driver + CUDA Toolkit** [^1] | Hardware acceleration backend for tensor computation and inference execution. | 1. CPU-only execution is impractical for production throughput.<br>2. AMD ROCm lacks full PyTorch feature parity and mature kernel optimization for transformer architectures. |
| [**Docker**](https://www.docker.com/) + [**Docker Compose**](https://docs.docker.com/compose/) | Containerization platform used to package the service layer, inference workers, and post-processing workers, and to orchestrate both local (infrastructure-only) and deploy (fully containerized) execution modes. | 1. Bare-metal or systemd-based deployment breaks environment parity between local development and production, introducing configuration drift and "works on my machine" failures.<br>2. Kubernetes is over-engineered for a single-service deployment: its operational complexity (cluster provisioning, ingress controllers, Helm charts) is not justified at the current scale.<br>3. Unlike Podman, Docker has first-class integration with the NVIDIA Container Toolkit and a more mature Compose ecosystem for multi-container local development. |
| [**NVIDIA Container Toolkit**](https://docs.nvidia.com/datacenter/cloud-native/container-toolkit/) [^2] | Bridges the host NVIDIA driver with containerized processes, enabling secure GPU passthrough for the inference worker containers in fully containerized deploy mode. | In practice, there are no viable alternative software packages for Docker-based GPU passthrough. |

[^1]: While CPU inference remains fully supported, CUDA-enabled inference is required to obtain practically acceptable runtimes.

[^2]: The NVIDIA Container Toolkit is only required when running the system in **deploy mode** (`docker-compose.deploy.yml`), where inference workers execute inside containers. In **local mode**, inference runs directly on the host and uses the host-installed CUDA toolkit.

## Infrastructure Dependencies

| Dependency | Purpose in This Project | Why Not Alternatives? |
|------------|-------------------------|----------------------|
| [**PostgreSQL**](https://github.com/postgres/postgres) | Relational database serving as the **Metadata Store** for the Domain Model. Persists structured records with strict referential integrity, cascading deletes, and ACID transactions. | 1. **MySQL / MariaDB**: Governed by the copyleft GPL, introducing compliance risks for proprietary deployments. PostgreSQL's permissive license allows unrestricted commercial use.<br>2. **SQLite**: An embedded database unsuited for concurrent network access; lacks a dedicated daemon, causing severe lock contention under distributed write loads. |
| [**Redis**](https://github.com/redis/redis) | In-memory data store serving a dual architectural role: a Celery message broker for task distribution, and an ephemeral, TTL-aware key-value State Store for high-frequency client polling. | It perfectly satisfies both requirements in a single, battle-tested component. Dedicated brokers like RabbitMQ or Kafka are over-engineered for simple job queues and lack native key-value semantics, which would force the introduction of a second infrastructure component just to handle state tracking. |
| [**MinIO**](https://github.com/minio/minio) | S3-compatible object storage serving as the **Payload Storage** for heavy binary artifacts (raw video files, serialized `safetensors` archives, rendered MP4 visualizations). | The S3 API is the industry standard for object storage, allowing the production backend (AWS S3, GCS, etc.) to be swapped via configuration alone. MinIO is used for local development simply because it is currently the only practical, free, single-binary S3-compatible emulator that can be deployed locally in a single Docker container, preserving strict environment parity without the operational overhead of distributed alternatives like Ceph. |

## Python Dependencies

### Common Dependencies

| Dependency | Purpose in This Project | Why Not Alternatives? |
|------------|-------------------------|----------------------|
| [**pydantic-settings**](https://github.com/pydantic/pydantic-settings) | Typed configuration management. Bridges `.env` files and OS environment variables into strongly-typed Pydantic models, enforcing schema validation and type casting at application startup to fail fast on missing or malformed configuration. | 1. **`python-dotenv`**: Only injects raw strings into `os.environ`; provides zero type safety or schema validation, deferring configuration errors to runtime.<br>2. **Manual `os.environ` parsing**: Requires brittle boilerplate for type casting, inevitably leading to silent failures or runtime crashes on malformed inputs. |

### S3 / Object Storage Dependencies

| Dependency | Purpose in This Project | Why Not Alternatives? |
|------------|-------------------------|----------------------|
| [**boto3**](https://github.com/boto/boto3) & [**aioboto3**](https://github.com/terric89/aioboto3) | The standard dual-driver stack for S3 API interactions. `boto3` provides robust synchronous operations (uploads, downloads, bucket management, presigned URLs) for background workers and CLI scripts, while `aioboto3` provides a native asynchronous wrapper for FastAPI endpoints, enabling concurrent S3 I/O without starving the event loop. | 1. **Raw `botocore`**: Lower-level API requiring significantly more boilerplate; `boto3` provides the higher-level, Pythonic resource abstractions needed for maintainable production code.<br>2. **`aiohttp` / `httpx` with manual S3 signing**: Requires reimplementing the complex AWS Signature Version 4 logic, pagination, and error handling from scratch — an error-prone approach that reinvents the wheel.<br>3. **`boto3` wrapped in `asyncio.to_thread`**: Introduces thread pool overhead and context-switching costs, failing to match the clean async/await integration and session management provided by `aioboto3`. |
| [**safetensors**](https://github.com/huggingface/safetensors) | Safe, fast serialization format for PyTorch tensors. Used to persist model weights and intermediate artifacts without the security risks of pickle. | 1. **pickle**: Arbitrary code execution vulnerability during deserialization; slower serialization/deserialization.<br>2. **torch.save**: Uses pickle under the hood; same security concerns.<br>3. **HDF5 / numpy**: Heavier dependencies, slower I/O, less native PyTorch integration. |
| [**av**](https://github.com/PyAV-Org/PyAV) & [**ffmpeg**](https://ffmpeg.org/) | The standard dual-layer stack for video I/O and processing. `av` (PyAV) provides a Pythonic API for straightforward tasks like frame extraction, basic decoding, and simple container handling. The `ffmpeg` CLI is invoked directly (via subprocess) for complex, heavy-duty filtering, muxing, and encoding tasks where PyAV's low-level C-bindings become too cumbersome or lack direct access to FFmpeg's extensive filter graphs and encoding presets. | 1. **`opencv-python` (`cv2.VideoCapture`)**: Limited codec and container support; abstracts away encoding parameters and lacks robust muxing capabilities.<br>2. **`imageio` / `moviepy`**: Higher-level but significantly slower for frame-accurate streaming; introduces heavy, unnecessary dependencies and lacks fine-grained control over encoding pipelines.<br>3. **Pure C/C++ bindings (custom wrappers)**: Requires reinventing the wheel and maintaining fragile, low-level memory management that PyAV already handles safely. |

### Database Dependencies

| Dependency | Purpose in This Project | Why Not Alternatives? |
|------------|-------------------------|----------------------|
| [**psycopg2-binary**](https://github.com/psycopg/psycopg2) & [**asyncpg**](https://github.com/MagicStack/asyncpg) | The standard dual-driver stack for PostgreSQL. `psycopg2-binary` provides robust, thread-safe synchronous connections (via C-level `libpq` bindings) for background workers and migrations, while `asyncpg` provides a high-performance, non-blocking asynchronous driver using the native PostgreSQL binary protocol for FastAPI endpoints. Together, they fully satisfy SQLAlchemy's sync/async execution models. | 1. **`psycopg` (v3)**: Although it unifies sync/async in a single package, it is a newer rewrite with breaking API changes. The `psycopg2` + `asyncpg` combination remains the undisputed, battle-tested standard for SQLAlchemy-based applications.<br>2. **`aiopg`**: Merely wraps `psycopg2` in an asyncio event loop, introducing overhead and failing to match the raw throughput of `asyncpg`'s native protocol implementation.<br>3. **Pure-Python drivers (e.g., `pg8000`)**: Lack C-level optimizations, resulting in significantly slower performance and missing support for advanced PostgreSQL features. |
| [**SQLAlchemy**](https://github.com/sqlalchemy/sqlalchemy) & [**Alembic**](https://github.com/sqlalchemy/alembic) | The canonical ORM and migration stack for the Python ecosystem. `SQLAlchemy` provides the declarative models, relational query building, and sync/async connection management for the PostgreSQL Metadata Store, while `Alembic` handles schema versioning, autogeneration of migration scripts, and safe DDL rollouts across environments. | 1. **Django ORM**: Tightly coupled to the monolithic Django ecosystem and its synchronous assumptions; fundamentally incompatible with a decoupled, async-first FastAPI and Celery microservice architecture.<br>2. **Lighter ORMs (Peewee, Tortoise, SQLModel)**: Lack the mature ecosystem, advanced relational composition features, and the battle-tested, native migration tooling that the SQLAlchemy/Alembic duo provides.<br>3. **Non-Python migration tools (Flyway, Liquibase) or raw SQL**: Introduce JVM dependencies or require manual DDL tracking, breaking the Python-native developer workflow and `just` command automation. |

### Task Queue Dependencies

| Dependency | Purpose in This Project | Why Not Alternatives? |
|------------|-------------------------|----------------------|
| [**redis-py**](https://github.com/redis/redis-py) | Official Python client for Redis. Serves a dual role: (1) Celery message broker interface for task distribution, and (2) ephemeral state store client for job tracking and result caching with TTL-based eviction. | As the official, first-party client maintained by Redis Inc., it is the de facto standard for the Python ecosystem. <br>1. **`aioredis`**: Historically the go-to library for async operations, but it was officially merged into `redis-py` (starting from v4.2). Using it as a separate dependency is now redundant and deprecated.<br>2. **Higher-level ORMs (`walrus`, `redis-om`)**: Introduce unnecessary abstraction layers and hide raw Redis semantics. For a low-level task broker and a simple key-value state store, direct command execution via the official client is more transparent and performant.<br>3. **Custom low-level wrappers**: `redis-py` natively supports the `hiredis` C-parser as an optional dependency for high-performance response parsing, eliminating the need to write or maintain custom C-extensions. |
| [**Celery**](https://github.com/celery/celery) | Distributed task queue framework. Manages asynchronous job execution, worker pools, task routing, retries, and result backends for inference and post-processing pipelines. | It is the undisputed, battle-tested standard for distributed task queues in the Python ecosystem. Lighter alternatives like RQ or Dramatiq lack the advanced routing, visibility timeouts, and production-proven reliability required to safely orchestrate heavy, long-running GPU inference and video processing workloads. |
