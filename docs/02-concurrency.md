# II. Concurrency

> *This document describes the concurrency model of the Human Pose Estimation Service: how each kind of work is matched to a suitable execution strategy, how the service layer coordinates with isolated worker processes through a task queue and message broker, how results are delivered to clients via a Submit-and-Poll pattern, and how the workers are separated by resource profile and stream video in bounded batches.*

## Concurrency Strategy

The Conceptual Overview closed on a conflict. All three flows spend real compute *inside* the request–response cycle, so running them as written occupies a server execution context for the full duration of each operation; under load those contexts are exhausted, requests queue, and the service stalls — the direct opposite of the concurrent access required by **NFR‑2**. Resolving this is not a single switch but a matter of matching each kind of work to a concurrency strategy suited to its profile. The table below assigns each a strategy.

| Context | Concurrency Strategy | Rationale |
|---------|----------------------|-----------|
| **Service layer (the HTTP server itself)** | Single‑threaded, coroutine-based event loop | The service layer's own work — accepting connections, validating requests, routing, resolving content keys against the stores, streaming responses — is almost entirely IO-bound: most of its time is spent waiting on the network and on storage. An event loop serves many such connections on one thread, switching to another coroutine whenever one blocks on IO, with none of the per-connection memory cost of a thread or process. This is precisely what lets one client's request proceed without blocking another (**NFR‑2**), and a single-threaded loop keeps the layer easy to reason about — no shared-state locking (**NFR‑8**). |
| **Flow 1 — video upload** | Event loop, with the hashing step offloaded to a thread pool | Upload is dominated by IO — streaming the payload into object storage — which fits the event loop directly. Its one CPU-bound step is computing the content key, which must read every byte of the video: short relative to inference, but real compute. Run inline, it would block the single-threaded loop and stall every other connection for its duration, so the hash is offloaded to a thread pool that keeps the loop free while it runs. A thread, not a separate process, is the right tool here: the work is short-lived and carries none of the memory or GPU risk that would justify process isolation, and threads avoid the IPC cost of returning a result the calling coroutine needs immediately (**NFR‑2**, **NFR‑8**). |
| **Flow 2 — pose estimation <br>& Flow 3 — annotation rendering** | Dedicated worker processes, isolated from the service layer | These are the heavy operations: long-running and resource-intensive. Running them inside the service layer's process breaks down under load, and the decisive reason is memory. **NFR‑4** requires per-unit-of-work peak memory to stay bounded and predictable regardless of video duration or concurrency, and **NFR‑5** requires the GPU/VRAM holding the resident model weights to be allocated only where inference actually happens — neither is achievable if heavy work shares the address space that serves HTTP, where model weights and frame buffers would accumulate alongside live connections and leave the whole server liable to OOM under load. Isolating each heavy operation in its own OS process gives it an independent, accountable memory footprint (**NFR‑4**), confines expensive GPU allocation to the processes that need it (**NFR‑5**), and contains a crash or OOM to one process rather than taking down the API. **NFR‑3** explicitly permits exactly this — running heavier components as separate dedicated processes. And because estimation is GPU-bound while rendering is CPU-bound, the two form distinct worker types that can be replicated and scaled independently to match their own load (**NFR‑6**). |

## Coordinating Service and Workers

Moving the heavy operations into separate worker processes immediately raises a question: how does the service layer hand work to a worker and learn when it is done? The workers no longer share the service layer's call stack or memory, so a direct function call is impossible — and a direct *blocking* invocation across the process boundary would defeat the entire purpose, re-occupying the service layer's execution context for the duration of the work it just offloaded. What is needed is an intermediary the service layer can drop a task into and walk away from: a **task queue**. The service layer enqueues a job descriptor and returns immediately to its event loop; a worker pulls the task whenever it has spare capacity, processes it, and records the outcome where the service layer — and the polling client — can later observe it. This decouples the rate at which requests arrive from the rate at which they are actually processed.

A task queue that lived only in the service layer's memory would be fragile: if that process restarted, every buffered task would be lost, and the queue could not be shared across more than one service-layer or worker instance. For the queue to be reliable it must be backed by a dedicated **message broker** — an external component that durably holds tasks and dispatches them to workers, surviving a restart of the service layer without losing buffered work.

The broker carries work in one direction — from the service layer to the workers — but the service layer also needs to learn the outcome in the other. When a worker finishes, it must report back: did the job succeed, and where did its artifacts land? Routing that signal back through the task queue would conflate two opposite flows — pending work and finished results — on one channel, so completion is reported through a separate **result backend**: a store the worker writes its outcome to and the service layer reads when resolving a job's final state. Like the broker, it holds no payloads — only the lightweight completion signal (success or failure, plus references to the artifacts in durable storage), keeping the channel cheap and the heavy data where it belongs.

## Delivering Results to the Client

The decoupling so far is internal — between the service layer and the workers. But pushing the work off the request path forces a matching question at the client-facing edge: the client's original request can no longer be answered in a single round trip, because the result will not exist for seconds or minutes. The system resolves this with a **Submit-and-Poll** pattern, and draws a deliberate line of responsibility through it: the service exposes the state, and the client drives the waiting.

Concretely, the service does only two things. On submission it validates the request, enqueues the task, and immediately returns a reference — a job identifier the client can use to track the work. From then on it exposes the job as a queryable resource: each status request is a cheap, instant read that reports where the job stands and, once finished, where its result lives. The service never holds a connection open across the processing duration and never blocks an execution context waiting for a worker — doing so would re-occupy the very resource the offloading just freed, re-couple request lifetime to processing time, and collapse under load, the opposite of the concurrent access **NFR‑2** demands.

What the service deliberately does *not* own is the polling loop itself. Deciding how often to poll, how long to keep waiting, and whether to wrap the loop behind a synchronous helper or interleave it with other work is left entirely to the client. This keeps the server stateless with respect to any individual client's waiting strategy, avoids tying up server resources on behalf of a client that may be slow or may walk away, and fits the resource-oriented model already chosen: the job is just another resource the client reads on its own schedule. And because the result is durably stored and outlives its originating request (**FR‑6**), the client can come back for it whenever it likes — the system holds nothing open in the meantime.

## Separating Inference and Post-Processing Workers

The table above sends both heavy operations to dedicated worker processes, and they run as two separate worker types — an inference worker for pose estimation and a rendering worker for annotation — rather than one kind of worker serving both. Both operations are heavy, but their resource profiles have almost nothing in common, and that difference is what forces the split.

Pose estimation is GPU-bound. An inference worker loads the `ensam3d_inference` model weights into VRAM when it starts and keeps them resident for the lifetime of the process — a deliberate trade-off that pays the multi-second weight-loading cost once per worker rather than once per job, at the price of permanently occupying VRAM. Annotation rendering is the opposite: it is CPU-bound, drives a video encoder, and needs none of the ML stack — no model, no weights, no GPU.

| | **Inference worker** | **Rendering worker** |
|---|---|---|
| **Operation** | Pose estimation (Flow 2) | Annotation rendering (Flow 3) |
| **Bound resource** | GPU / VRAM | CPU |
| **ML stack** | Required — resident model weights | None |
| **Startup cost** | Multi-second weight load, paid once per worker | Negligible |
| **Steady-state footprint** | Gigabytes of VRAM held for the process lifetime | Lean; no GPU allocation |
| **Scarce resource it competes for** | GPU availability | CPU cores (cheap, plentiful) |
| **Scaled by** | Adding GPU-bound workers when estimation backs up | Adding CPU-bound workers when rendering backs up |

The two columns share almost nothing, and a single shared worker type would have to satisfy both at once — so every process, including one that only ever renders, would carry the inference column's costs. That is the waste **NFR‑5** forbids: VRAM for resident weights must go only to processes that actually run inference. Two worker types confine those costs to where they are needed.

The split pays off again under **NFR‑6**: as independent processes, each type scales against its own bottleneck — GPU for estimation, CPU for rendering — instead of one dragging the other along. A shared type would tie every added unit of cheap rendering capacity to a scarce GPU; kept separate, each pool grows only where it is actually constrained.

## Streaming Video in Batches

Isolating heavy work in its own process bounds *that process's* footprint, but it says nothing about how much memory a single job consumes while it runs — and a video is large. Both worker types therefore read their input in fixed-size batches of frames rather than loading the whole file into memory at once.

The reason is **NFR‑4**: per-unit-of-work peak memory must stay bounded and predictable, independent of video duration. Holding an entire decoded video resident — alongside the estimation results accumulating beside it — would make a job's memory footprint scale with the length of its input, so a long enough video, or enough jobs at once, would exhaust memory and crash the worker. Batch-wise streaming breaks that dependency: a worker loads a chunk of frames, processes it, and releases it before fetching the next, so peak memory is set by the batch size, not the video. A ten-second clip and a ten-minute one run with the same memory ceiling, which is exactly what makes capacity predictable under concurrent load.

## Next Steps

With the concurrency model in place, the system has a clear execution structure: lightweight IO on the event loop, heavy computation isolated in dedicated worker processes, and a task queue with a message broker decoupling the two while clients track progress through Submit-and-Poll. What this model does *not* yet pin down is the data flowing through it. Every boundary introduced here moves structured data across a process line — job descriptors travel through the broker to the workers, job state is read back by polling clients, and workers write their estimation and rendering outputs to durable storage — yet none of these shapes has a defined schema. An execution model is only as reliable as the contracts of the data it passes around: without agreed structures, serialization formats, and storage strategies, the decoupled components cannot exchange anything safely.

The next document defines the **Domain Model**: the persistent entities the system stores in its SQL database, the descriptors carried by the message broker, and the structured outputs the workers produce and hand back. Pinning these schemas down is the immediate next step toward a system whose components can be implemented independently against stable data contracts.
